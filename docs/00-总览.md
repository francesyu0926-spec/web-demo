# 观点科技电子招投标交易平台 — 后端设计总览

> 配套文档：[01-数据库设计](./01-数据库设计.md) · [02-接口设计](./02-接口设计.md)
> 技术栈：Spring Boot 3.x + MySQL 8 + Spring Security(JWT) + MyBatis-Plus + Redis + MinIO

---

## 1. 系统定位

面向政府/企业的电子招投标交易平台，覆盖**招标—报名—开标—评标—定标公示**全流程，多角色协同。

## 2. 角色模型

| 角色码 | 名称 | 职责 |
|---|---|---|
| GUEST | 游客 | 浏览招标公告、找招标/找中标、查看指南法规（无需登录） |
| BIDDER | 投标人(供应商) | 报名、递交投标文件、缴保证金、查看我的报名 |
| AGENT | 项目经理(招标代理) | 建项目、审核报名、抽专家、开标、汇总、定标 |
| EXPERT | 评标专家 | 确认/签到、下载投标文件、五类评审打分 |
| ADMIN | 平台管理员 | 专家审核、内容维护、平台监管 |

> 一个用户可拥有多个角色（如既是投标人又是专家），通过 `POST /api/auth/switch-role` 切换激活角色。

## 3. 项目状态机（后端流程主线）

```text
        publish            (到报名截止)        open
DRAFT ──────────▶ PUBLISHED ──────────▶ REG_CLOSED ──────▶ OPENED
草稿              报名中               报名截止           开标
                                                            │ 进入评标
                                                            ▼
                                                       EVALUATING ──award──▶ WIN_PUBLICITY ──finish──▶ FINISHED
                                                       评标中               中标公示                 结束

任意阶段 ──abort──▶ ABORTED(废标/流标)
```

**状态约束（写接口必须校验）**

| 操作 | 允许的前置状态 | 角色 |
|---|---|---|
| 编辑项目 | DRAFT | AGENT |
| 发布 | DRAFT → PUBLISHED | AGENT |
| 报名 | PUBLISHED | BIDDER |
| 递交投标文件 | PUBLISHED 且报名已通过 | BIDDER |
| 报名审核 | PUBLISHED / REG_CLOSED | AGENT |
| 抽取专家 | REG_CLOSED | AGENT |
| 开标 | REG_CLOSED → OPENED | AGENT |
| 下载投标文件 | OPENED 之后 | EXPERT/AGENT |
| 评分 | EVALUATING 且专家已签到 | EXPERT |
| 定标公示 | EVALUATING → WIN_PUBLICITY | AGENT |
| 结束 | WIN_PUBLICITY → FINISHED | AGENT |

> 建议封装 `TenderStateMachine`，在 Service 层统一做 `assertStatus(project, allowed...)`，禁止乱序操作。

| 功能 | GUEST | BIDDER | AGENT | EXPERT | ADMIN |
|---|:--:|:--:|:--:|:--:|:--:|

| 建项目/审核报名 | — | — | ✅ | — | — |
| 抽专家/开标/定标 | — | — | ✅ | — | — |

> 落地用 Spring Security `@PreAuthorize("hasRole('AGENT')")`，并在数据层做归属校验（如代理只能操作自己的项目）。

## 5. 关键技术点

1. **鉴权**：JWT 携带 `userId + activeRole`；切角色重新签发。Redis 存验证码、token 黑名单。
2. **评分汇总**：最终得分 = Σ(各评审项专家平均分 × 权重)；形式/资格/响应性为"通过制"(任一不通过则淘汰)，商务/技术为"打分制"。
3. **投标文件保密**：`OPENED` 之前任何人（含代理）不可下载投标文件，下载接口强校验项目状态。
4. **合规审计**：发布、开标、评分、定标等关键动作写 `operation_log`。
5. **待定功能**：保证金、发票、专家电子签名（原型标注"待定/暂缓"）——表与接口先预留，开关控制是否启用。

## 6. 建议的开发顺序（迭代）

1. **基建**：项目骨架 + 统一响应/异常 + Security(JWT) + Knife4j 接口文档。
2. **用户域**：注册/登录/角色切换 + 投标人资料 + 专家申请审核。
3. **招标域**：建项目/发布 + 游客找招标找中标 + 公告。
4. **投标域**：报名 + 报名审核 + 递交投标文件 + 文件上传(MinIO)。
5. **开评标域**：抽专家 + 签到 + 评分 + 汇总 + 定标公示（核心难点）。
6. **通用域**：消息通知、内容、投诉、操作日志。

## 7. 与设计稿的对应

- 角色与页面分组同 CoDesign 原型 / 蓝湖 UI（投标人 / 项目经理 / 专家 / 游客）一致，可逐页对照接口。
- 蓝湖"我的报名"表格 → `GET /api/my/registrations`；门户首页公告区 → `GET /api/announcements`、`GET /api/tenders`。
