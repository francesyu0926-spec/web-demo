## 1. 系统定位

面向公共资源交易的电子招投标交易平台（建设单位：山西观点科技有限公司），覆盖
**项目注册 → 公告发布 → 报名缴费 → 投标递交（加密）→ 在线开标解密 → 评标评审 → 中标公示 → 合同/归档**
的全流程闭环，遵循《电子招标投标办法》等规范，强调全程留痕、电子签章、智能清标。

## 2. 角色模型（依据需求文档修正）

> ⚠️ 重要修正：原设计把"项目经理=招标代理"，需求文档中 **项目经理与招标人是不同角色**。

| 角色码 | 名称 | 职责 |
|---|---|---|
| GUEST | 游客 | 浏览公告、找招标/找中标、查看指南法规、提建议（部分功能需登录） |
| BIDDER | 投标人(供应商) | 报名、缴费、递交并加密投标文件、解密、谈判/二轮报价回复、缴代理费、查看我的报名/项目 |
| TENDERER | 招标人 | 被项目经理邀请成为招标人，关联到具体招标项目；查看本方项目进展 |
| MANAGER | 项目经理 | 平台业务运营角色：创建/发布项目、报名审核、编辑评审表、抽取专家、推进开评标、发布中标公示、推送/管理代理费、归档 |
| EXPERT | 评标专家 | 确认/拒绝邀请、签到、下载文件、形式/资格/响应性/商务/技术/报价评审、谈判与二轮报价发起、评标报告签名 |
| SALES | 业务员 | 平台已注册业务员（需求文档仅列名，功能待细化） |
| ADMIN | 平台管理员 | 后台管理与运营、专家/角色申请审核、内容维护、监管 |

- 一个用户可拥有多个角色，通过 `POST /api/auth/switch-role` 切换激活角色（"我的"界面账号身份切换）。
- 项目经理隶属**代理公司**（`agency_company`，"我的代理公司"）；项目经理可**邀请招标人**（`tenderer_invite`）。
- 角色申请：招标人可申请成为 **项目经理 / 专家**（`role_application`，需审核）。

## 3. 采购方式（procurement_type）

| 类别 | 取值 | 说明 |
|---|---|---|
| 招标方式 | `PUBLIC` 公开招标 | 公告邀请不特定主体，竞争充分 |
| 招标方式 | `INVITE` 邀请招标 | 邀请≥3家，不发公告 |
| 非招标方式 | `INQUIRY` 询价 | |
| 非招标方式 | `SINGLE` 单一来源 | |
| 非招标方式 | `NEGOTIATION` 竞争性谈判 | 与≥3家分别谈判、允许二次报价，不评分看价格 |
| 非招标方式 | `CONSULTATION` 竞争性磋商 | 与谈判类似，但需**评分**确定成交候选人 |

> 采购方式决定后续流程：**竞争性谈判/磋商** 会触发"谈判磋商 + 二轮报价"环节。

## 4. 项目状态机（双维度：项目状态 + 评审节点）

需求文档将项目状态与评审节点分开管理：

**项目状态 `tender_project.status`**
```text
DRAFT草稿 → BIDDING投标中 → OPENING开标中 → OPENED已开标 → AWARDED已定标(中标) → FINISHED已结束 → ARCHIVED已归档
                                                                          ↘ ABORTED废标/流标
```

**评审节点 `tender_project.eval_node`**
```text
NOT_STARTED未开始 → REVIEWING评审中 ↔ NEGOTIATING谈判磋商待回复 ↔ SECOND_QUOTE二轮报价待回复 → FINISHED已结束
```

**关键阶段动作（写接口必须校验状态 + 角色）**

| 阶段 | 触发/动作 | 角色 | 前置 |
|---|---|---|---|
| 发布项目 | 创建并提交（解析招标文件自动带入字段） | MANAGER | DRAFT→BIDDING |
| 报名 | 获取招标文件→提交报名 | BIDDER | BIDDING |
| 报名审核 | 通过/驳回 | MANAGER | BIDDING |
| 报名缴费 | 文件费+平台使用费 → 报名成功 | BIDDER | 审核通过 |
| 递交投标文件 | 递交→加密(六位密码)，可撤回重交 | BIDDER | BIDDING & 报名成功 |
| 编辑评审表 / 抽取专家 / 发布变更 / 跳过评审 | | MANAGER | BIDDING |
| 开标 | 到开标时间进入开标中 | 系统/MANAGER | BIDDING→OPENING |
| 专家签到 → 确认签到 | 专家签到，经理确认 | EXPERT/MANAGER | OPENING |
| 文件解密 | 投标人签字解密(六位密码+签名) | BIDDER | 专家已签到 |
| 确认报价 / 符合性审查 | 形式/资格/响应性 | EXPERT | 已解密 |
| 谈判磋商 / 二轮报价 | 组长发起，投标人回复 | EXPERT/BIDDER | 评审中(竞谈/竞磋) |
| 商务/技术/报价评分 → 评审得分 | | EXPERT | 评审中 |
| 评标报告：生成→编辑→推送→签名→导出 | | MANAGER/EXPERT | 评审节点结束 |
| 发布中标公示 | 默认排名第一，可多家 | MANAGER | 评审节点FINISHED |
| 推送代理费(固定/标准折扣) | | MANAGER | AWARDED |
| 缴纳代理费 | | BIDDER(中标人) | 代理费已推送 |
| 发布中标通知书 | 缴费后发布，首页显示 | MANAGER | 代理费已缴 |
| 项目归档 | 归档后仅经理/管理员可见 | MANAGER | FINISHED |

> 建议封装 `TenderStateMachine`，在 Service 层统一 `assertStatus(project, allowed...)` 与 `assertEvalNode(...)`，杜绝乱序。

## 6. 关键技术点

1. **鉴权**：JWT 携带 `userId + activeRole`，切角色重签；Redis 存短信验证码、热门搜索计数、token 黑名单。
2. **投标文件加解密**：投标人递交后**六位密码加密**，开标前任何人不可查看；专家评审前需对应投标人**签字解密**（六位密码+电子签名）。下载接口强校验项目状态与解密状态。
3. **缴费体系**：报名缴费（文件费 + 平台使用费，两栏分算）、代理费；支付方式 微信 / 支付宝 / 对公汇款；生成账单/订单明细。
4. **评分模型（v0603）**：6 类评审 = 形式/资格/响应性（通过制，**全体专家**）+ 商务/技术/报价（打分制）。**技术分**：全体专家打分取平均；**商务分/报价分**：**仅专家组长**打分（须先 `POST .../elect-leader` 推选组长）。最终得分 = 三类分数相加并排名。报价评分有 3 种固定计算方式（默认报价分 30）。
5. **评标报告**：随评审阶段逐步生成文档（未生成→待签名→已完成），经理编辑→推送至专家→专家签名/一键签名→经理一键导出。
6. **合规审计**：发布、开标、解密、评分、签名、定标等关键动作写 `operation_log`。
7. **待补充（需求文档标注）**：招采项目AI审查、保函业务、CA业务、我的中标、商机项目、保证金、业务结算（佣金/发票/项目结算）——表与接口先预留、开关控制启用。

## 7. 非功能需求（摘要）

- **性能**：最大同时在线 >10；系统/数据库可用率 ≥99.9%；查看页 1~2s、办理页 2~3s、统计页 5~10s。
- **灵活性**：可水平扩展、模块可拆分组合、按角色授权、提供对外 API 对接。
- **架构**：微服务 + 容器化 + 负载均衡，高并发动态组网。
- **数据**：分级分类存储、全流程治理、实时备份。
- **终端**：Win10/11；Chrome、360 等浏览器。

## 8. 建议开发顺序

> 完整功能对照见 [03-功能清单v0603](./03-功能清单v0603.md)（80 项，v0603-1）。

1. 基建：骨架 + 统一响应/异常 + Security(JWT) + Knife4j。
2. 用户域：注册/登录(账号/微信/手机)/角色切换 + 个人/企业资料 + 角色申请审核 + 代理公司/邀请招标人。
3. 招标域：创建/发布项目（招标文件解析带入）+ 公告 + 首页找招标找中标 + 搜索(历史/热门)。
4. 投标域：报名 → 审核 → 缴费 → 递交投标文件 → 加密/撤回/解密。
5. 开评标域（核心难点）：抽专家 → 签到 → 解密确认 → 符合性审查 → 谈判/二轮报价 → 评分 → 评审得分 → 评标报告 → 中标公示 → 代理费 → 中标通知书 → 归档。
6. 通用域：消息通知、内容(指南/法规/案例/下载/动态)、投诉建议、操作日志。
